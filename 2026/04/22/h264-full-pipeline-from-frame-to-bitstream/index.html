H.264 全流程讲解，从一帧图像到最终码流

H.264 全流程讲解，从一帧图像到最终码流

这篇文章按一条完整主线来讲清楚 H.264：输入到底是什么，编码时到底做了什么，输出到底是什么，I/P/B 帧如何协作，编码器和解码器为什么都必须缓存历史帧，以及 NALU 码流在整条链路中的位置。

H.264 最容易让人困惑的地方，不是概念名词多，而是链路长。宏块、帧内预测、运动估计、残差、变换、量化、熵编码、NALU、GOP、参考帧缓存，这些点单独看都不难，难的是把它们连成一条完整数据路径。

如果把问题压缩成一句话，H.264 本质上是在做这件事：尽量不直接保存整张图，而是想办法只保存“和已有画面相比，发生了什么变化”。为了做到这一点，编码器和解码器必须共享同一套参考画面，因此它们两边都离不开缓存机制。

一、先抓住主线：H.264 到底在压什么

H.264 编码器接收的不是 JPG、PNG，也不是 MP4 文件，而是未经压缩的原始视频帧，常见形式是 YUV420。你可以把它理解成一张张完整的裸像素图片。

编码器的目标不是“把图片换个格式保存”，而是把这张原始图像拆解成更容易压缩的信息：

  • 哪些区域和已有画面很像
  • 哪些区域只是发生了位置移动
  • 哪些区域才是真正新增的信息
  • 哪些高频细节可以被适当丢弃

最后输出的也不是“图片”，而是一串压缩后的视频比特流。这串比特流里包含参数信息、预测信息、残差数据和编码后的语法元素，最终再以 NALU 的形式组织起来。

可以先看整条编码主线。

flowchart LR
    A[Raw frame YUV] --> B[Split into MB]
    B --> C[Choose frame type]
    C --> D[Intra or inter prediction]
    D --> E[Residual]
    E --> F[Transform]
    F --> G[Quant]
    G --> H[Entropy coding]
    H --> I[NALU bitstream]

这张图里最关键的分水岭是 Intra or inter prediction。这里决定了当前帧到底是靠自己压，还是靠历史参考帧一起压。

二、一帧图像进入 H.264 后，到底经历了什么

1. 输入是一帧原始 YUV 图像

编码器看到的输入，本质上是一帧完整图像。假设是 1920×1080 的 YUV420 帧，那么这一帧里已经包含了亮度和色度采样信息，只是还没有经过压缩。

这一步的特点是：

  • 数据量大
  • 画面完整
  • 每个像素都真实存在
  • 还没有任何“只记录变化”的思想

也正因为原始帧数据太大，视频编码的核心任务就是把这类数据压到足够小。

2. 先把整张图切成宏块

H.264 早期编码分析常从 16×16 宏块 入手，这也是最常见的理解入口。虽然标准内部还有更细的子块划分和不同分区方式，但从学习主线来说，可以先把整张图理解为被切成很多个小块。

切块的目的很直接：

  • 便于做局部预测
  • 便于做局部运动搜索
  • 便于做局部残差压缩
  • 便于让不同区域采用不同编码策略

也就是说，H.264 不是拿整张图一次性做压缩，而是按块处理。

3. 判断当前帧是什么类型

编码器接下来要决定当前帧按什么方式编码，常见就是 I、P、B 三类。

帧类型 依赖关系 主要特点
I 帧 不依赖其他参考帧 压缩率较低，但独立性最强
P 帧 参考之前的参考帧 只保存相对过去的变化
B 帧 可参考前后参考帧 压缩率更高，但时序更复杂

这一步本质决定了后面的预测方式。

4. 做预测，找到“已有的信息”

预测是 H.264 最核心的一步，因为它决定了后续到底还剩多少东西需要编码。

4.1 I 帧做帧内预测

I 帧不参考历史帧，它只看当前帧内部已经编码过的邻近像素块，尝试预测当前块大概长什么样。

例如，一个区域如果本来就是平滑渐变或者明显边缘延续，那么周围像素就能给出相当好的估计，当前块就不必完整保存。

可以把 I 帧理解成：

  • 不向前借画面
  • 不向后借画面
  • 只从当前帧内部借上下左右邻域信息

4.2 P 帧做帧间预测

P 帧会去参考列表里找历史参考帧，然后问一个问题：

当前这个块，能不能在过去的某一帧里找到一个非常像的块？

如果能找到，编码器只需要记录：

  • 它来自哪一帧
  • 它在那一帧里的哪个位置
  • 当前位置和参考位置之间的偏移量，也就是运动矢量 MV

这就是运动估计和运动补偿背后的直觉。

flowchart LR
    A[Current block] --> B[Search in reference frame]
    B --> C[Best matched block]
    C --> D[Motion vector]
    D --> E[Pred block]

4.3 B 帧做双向预测

B 帧比 P 帧更进一步。它不只看过去，还可以同时参考前后参考帧。

更准确地说，B 帧可以参考前向和后向参考图像，编码和解码中通常伴随显示顺序与编码顺序的重排序。因此它不是一句简单的“直接读取未来图”就能概括完的，但从学习角度可以先把它理解成“双向借图”。

B 帧的好处是压缩率更高，代价是：

  • 编码逻辑更复杂
  • 参考关系更复杂
  • 缓存更多
  • 延迟更高

所以低延迟场景，比如实时互动直播，常常会尽量少用 B 帧，甚至不用。

5. 预测完后，只编码残差

当编码器拿到了预测块之后，当前块就不需要再“完整保存”了。真正要编码的是：

原始块 - 预测块 = 残差块

如果预测很准，残差就很小，甚至大部分系数都接近 0。这就是视频压缩能做得这么高效的根本原因之一。

flowchart LR
    A[Original block] --> C[Residual]
    B[Pred block] --> C[Residual]

这一步之后，编码器已经把“画面”转换成了“误差”。

6. 对残差做变换

残差还在空间域里，不够利于后续压缩。于是编码器会对残差块做整数变换，你可以把它理解成工程化实现的 DCT 思想：把像素变化重新表示成不同频率分量的组合。

变换之后通常会出现一个很有利于压缩的现象：

  • 低频能量更集中
  • 高频分量更容易变小
  • 很多高频细节在视觉上并不重要

因此，后面的量化就有了发力空间。

7. 量化，主动丢掉一部分细节

量化是有损压缩最直接的一步。它会把一些较小、较不重要的频率系数压得更粗糙，甚至直接变成 0。

这一步的效果通常是：

  • 码率明显下降
  • 大量系数归零
  • 视觉质量略有损失
  • 压缩效率大幅提高

如果把预测理解成“尽量少保存变化”，那量化就是“即便有变化，也不是每一点变化都必须保留得那么精确”。

8. 熵编码，把统计冗余进一步压干

经过量化之后，数据里通常已经有很多 0，也出现了明显的统计规律。这时候再用熵编码继续压缩。

H.264 常见两类熵编码方式：

  • CAVLC
  • CABAC

前者更直观，后者压缩效率更高，但实现和计算更复杂。

这一步做完之后，块级信息已经被编码成更紧凑的二进制语法元素。

9. 打包成 NALU，形成 H.264 比特流

编码器最终会把参数集、图像数据和其他辅助信息组织成 NALU。这也是应用层最常看到的 H.264 数据组织单位。

flowchart LR
    A[H264 stream] --> B[SPS]
    A --> C[PPS]
    A --> D[IDR slice]
    A --> E[P slice]
    A --> F[B slice]
    A --> G[SEI]

其中常见内容包括：

  • SPS：序列级参数，比如分辨率、Profile、Level 等
  • PPS：图像级参数
  • Slice：真正的帧压缩数据
  • SEI：补充增强信息

如果是 Annex B 形式的裸码流，NALU 之间通常通过起始码分隔，比如 0x000001 或 0x00000001。

到这里，编码器输出的就是 H.264 比特流，而不是“又一张压缩图片”。

三、H.264 最终输出到底是什么

这个问题特别容易和 MP4、FLV、MKV 混在一起。

H.264 本身输出的是视频编码后的比特流，也就是一串按标准语法组织起来的二进制数据。它本身不是封装格式，也不是播放器直接展示出来的图像。

因此要分清三层：

  1. 原始视频帧，比如 YUV
  2. H.264 编码后的码流，比如 Annex B 比特流
  3. 容器封装格式，比如 MP4、FLV、MKV

三者关系可以画成下面这样。

flowchart LR
    A[Raw YUV frames] --> B[H264 encoder]
    B --> C[H264 bitstream]
    C --> D[MP4 FLV MKV container]

也就是说：

  • H.264 负责压缩视频内容
  • MP4 负责把音视频、时间戳、索引等内容装进一个容器里
  • 播放器最终还是要先把 H.264 码流解出来，才能重建出 YUV 帧

一句话说清楚就是：H.264 的最终输出是比特流，这个比特流不能直接显示，必须经过解码器重建为图像帧后才能播放。

四、解码端在做什么，本质上就是把上面的过程倒回来

解码不是“把文件直接显示出来”，而是沿着编码的反方向把画面一步步恢复出来。

flowchart LR
    A[H264 bitstream] --> B[Parse NALU]
    B --> C[Entropy decode]
    C --> D[Inverse quant]
    D --> E[Inverse transform]
    E --> F[Residual]
    F --> G[Prediction and MC]
    G --> H[Reconstructed frame]
    H --> I[Deblocking]
    I --> J[Output YUV]

1. 先解析 NALU

解码器首先把输入比特流按 NALU 拆开，识别出 SPS、PPS、Slice 等内容，建立必要的解码上下文。

2. 熵解码

把 CABAC 或 CAVLC 编出来的语法元素还原回来，得到量化后的系数、运动信息、模式信息等。

3. 反量化和反变换

这两步把压缩域的数据往像素域方向拉回来，恢复出残差块。

4. 再做预测和运动补偿

这里和编码器必须严格一致。

  • I 帧按帧内预测规则恢复
  • P/B 帧按参考帧和运动矢量做运动补偿

最后得到：

预测块 + 残差块 = 重建块

这才真正接近原始画面。

5. 去块滤波后输出重建帧

由于块编码和量化，重建图像边界上可能出现块效应。H.264 会通过去块滤波进一步改善视觉质量。处理完之后，解码器才能输出最终的重建 YUV 帧给播放器或后处理模块。

五、为什么 H.264 一定要缓存历史帧

这是很多人第一次学视频编码时最想追问的问题，也是最关键的工程直觉之一。

答案很明确：一定要缓存，而且编码器和解码器两边都要缓存。

因为 P 帧和 B 帧不是独立存在的，它们必须依赖参考图像。如果没有缓存，预测就无从谈起。

1. I 帧为什么相对独立

I 帧不依赖其他参考帧，所以它可以作为随机访问入口。你从一个 I 帧甚至 IDR 帧开始，后续才有可能逐步恢复出完整视频。

但要注意，I 帧独立并不代表它编码后没用了。恰恰相反，I 帧编码并重建完成后，通常会进入参考帧缓存，供后续 P/B 帧使用。

2. P 帧为什么必须读缓存

P 帧的核心思想就是“参考过去”。

如果当前画面里有一个人从左往右移动，编码器不会重新保存整个人，而更可能记录：

  • 这个块可以从某个参考帧里找到
  • 它向右移动了多少像素
  • 剩余没预测准的部分作为残差补进去

这一步离不开参考帧缓存。

3. B 帧为什么对缓存和时序要求更高

B 帧可以同时参考前后参考图像，所以它带来的问题不只是“多存一帧”，而是：

  • 参考管理更复杂
  • 编码顺序和显示顺序可能不一样
  • 解码器要等到足够参考信息准备好
  • 整体延迟更高

因此 B 帧常见于更关注压缩率的场景，而不是极致低延迟场景。

六、缓存里存的到底是什么，不是原始帧，而是重建帧

这是一个很容易在面试里被追问的点。

编码器缓存的不是原始输入帧，而是重建帧。解码器缓存的也是重建帧。

为什么一定要这样？因为编码器和解码器后面做预测时，必须看到完全一致的参考图像。如果编码器拿原始图做参考、解码器拿重建图做参考，那么两边预测结果就会逐步漂移，最终完全对不上。

所以编码器内部通常会有一条本地重建路径。

flowchart TD
    A[Original frame] --> B[Prediction]
    B --> C[Residual]
    C --> D[Transform and quant]
    D --> E[Entropy coding]
    D --> F[Inverse quant and inverse transform]
    F --> G[Add pred]
    G --> H[Reconstructed frame]
    H --> I[Reference buffer]

这条路径说明了一个特别重要的事实：

  • 编码器一边在生成输出码流
  • 一边也在本地“模拟解码”
  • 模拟出来的重建帧会进入参考缓存
  • 之后的 P/B 帧预测都基于这些重建帧进行

这就是为什么很多资料会说：编码器内部其实也有一个小型解码环。

七、IDR 帧为什么像一次“缓存重置”

普通 I 帧强调的是“这一帧本身可以独立编码”，但在工程使用中，大家更常强调的是 IDR 帧。

IDR 可以理解成一次更强的刷新点。它的意义是：

  • 从这个点开始，后续参考关系不再依赖它之前的画面
  • 解码器可以安全丢弃旧参考缓存
  • 丢包、误码、漂移不容易继续向后扩散

所以在很多资料里会把它概括成一句很实用的话：

IDR 帧会让参考链路重新开始，相当于一次缓存重置。

这也是播放器 seek、直播恢复、丢包后重新对齐时非常关键的一个点。

八、用 GOP 把 I/P/B 帧关系串起来

学习帧类型时，最容易记住的是 GOP，也就是图像组。

一个典型的 GOP 里，I、P、B 帧大致会形成下面这种结构。

flowchart LR
    A[IDR] --> B[B]
    B --> C[B]
    C --> D[P]
    D --> E[B]
    E --> F[B]
    F --> G[P]
    G --> H[Next IDR]

如果从显示顺序去看，可以把它粗略理解为：

I -> B -> B -> P -> B -> B -> P -> IDR

但在真正编码和传输时，为了让 B 帧获得它需要的参考图像，编码顺序往往会发生重排序，所以学习时要区分：

  • 显示顺序，播放器看到的顺序
  • 编码顺序 / 传输顺序，码流里实际组织和送出的顺序

这也是 B 帧比 P 帧更复杂的一个核心原因。

九、如果只背一版面试答案，应该怎么说

可以把整条链路压缩成下面这段。

1. 编码主线

H.264 编码时，先把原始 YUV 帧按块划分，然后根据帧类型做帧内预测或帧间预测。预测后只编码残差，再对残差做变换、量化和熵编码，最后打包成 NALU 输出成 H.264 比特流。

2. 解码主线

解码时先解析 NALU，再做熵解码、反量化、反变换，恢复出残差，然后结合帧内预测或运动补偿重建图像，经过滤波后输出重建帧。

3. 缓存主线

H.264 的 P/B 帧必须依赖参考图像，所以编码器和解码器两边都要缓存参考帧。缓存的不是原始输入帧，而是重建帧。IDR 帧会重启参考链，常被视为一次缓存刷新点。

十、最后把这件事真正想明白

如果从工程角度重新看一遍，H.264 做的事情其实非常统一：

  • 用预测消灭时间和空间上的重复信息
  • 用残差只保留没预测准的部分
  • 用变换和量化压缩视觉上不重要的细节
  • 用熵编码榨干统计冗余
  • 用 NALU 组织成标准码流
  • 用参考帧缓存保证编码器和解码器看到同一条历史画面链

所以，H.264 不是“把每一帧单独压成更小的图片”，而是把一串连续画面之间的关系编码进了码流里。

真正理解了这一点，I/P/B 帧、GOP、运动估计、参考帧缓存、IDR 刷新这些概念就会自然连起来。
