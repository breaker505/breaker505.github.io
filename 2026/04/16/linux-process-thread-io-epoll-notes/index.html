从 task_struct 到 epoll，系统梳理 Linux 进程、线程与 I/O 核心知识

Linux 这一块最容易学散。

一会儿在看 task_struct，一会儿在背文件描述符，一会儿又跳到 socket、epoll、NAPI，最后脑子里剩下一堆零散关键词，却很难形成真正的系统图。

这篇文章不准备把所有知识点都堆成词条，而是尝试把 进程 / 线程、文件与 I/O、socket、epoll 串成一条主线。你可以把它当成一篇偏“体系化复盘”的 Linux 知识整理。

说明：本文主要站在 Linux 内核常见实现和工程理解视角来讲，个别地方会做适度抽象，方便建立整体认知。

一、先建立主线：Linux 到底在管理什么

如果从内核视角看，Linux 管理的核心对象其实可以粗暴概括成三类：

  • 执行实体，也就是 task
  • 内存地址空间
  • I/O 对象与等待机制

往后面拆，你会发现很多看似分散的问题，本质上都在围绕这几件事：

  • 一个任务怎么被调度
  • 多个任务怎么共享或隔离资源
  • 用户态如何通过文件描述符访问内核对象
  • 数据没准备好时，线程该怎么等
  • 多个 fd 同时存在时，如何避免低效轮询

所以本文会按下面这条线展开：

  1. 进程与线程的本质模型
  2. 文件描述符与 Linux I/O 抽象
  3. 阻塞、非阻塞与 I/O 多路复用
  4. epoll 的核心设计

二、进程与线程：在 Linux 里本质上都是 task

很多教材会先严格区分“进程”和“线程”，但在 Linux 内核实现里，更底层的视角其实是：

进程和线程本质上都对应一个 task_struct。

也就是说，内核真正调度的，是 task，而不是你课本里那个抽象的“进程”概念。

flowchart TD
    A[task_struct] --> B[mm_struct 地址空间]
    A --> C[files_struct 文件描述符表]
    A --> D[signal_struct 信号]
    A --> E[thread_struct 寄存器上下文]

2.1 task_struct 里到底有什么

task_struct 很大，真实内核源码里成员很多，不可能面试时全背下来。更实用的做法是：

先按功能分组理解。

下面给一个“面试和体系理解够用”的简化版：

你可以把它理解成：

task_struct 描述了“一个可被调度执行的任务”，以及这个任务当前的状态、资源和执行上下文。

2.1.1 先抓最核心的几类成员

如果是面试或者复盘，我建议优先记下面几组：

1. 调度相关

比如：

  • state
  • prio
  • static_prio
  • sched_entity

这些决定了一个 task 当前处于什么状态、如何参与调度。

2. 标识相关

比如：

  • pid
  • tgid
  • parent
  • children

这组字段帮助内核知道：

  • 这个任务是谁
  • 它属于哪个线程组
  • 它的父子关系是什么

3. 内存相关

最重要的是：

  • mm
  • active_mm

这组字段决定了这个任务用的是哪份地址空间。

4. 文件相关

最重要的是：

  • files
  • fs

前者和文件描述符表相关，后者和当前工作目录、root 目录等相关。

5. 执行上下文相关

比如：

  • stack
  • thread

它们关系到 task 被切走、再切回来时，CPU 上下文怎么恢复。

2.2 Linux 为什么说“不严格区分进程和线程”

关键不在 task_struct 本身，而在于 多个 task 是否共享资源。

最核心的区分点之一，就是：

  • 是否共享 mm_struct
  • 是否共享 files_struct
  • 是否共享信号处理结构等

其中，mm_struct 特别关键，因为它描述的是用户态地址空间。

如果两个 task_struct 指向同一个 mm_struct，那么它们共享同一份用户地址空间，这在工程上就可以近似理解为：

它们属于同一个进程里的多个线程。

如果各自拥有独立的 mm_struct，那它们更接近我们平常说的“不同进程”。

2.3 mm_struct 为什么这么关键

mm_struct 里最值得先记住的是这些：

它管理的是整个用户态虚拟地址空间的布局。

所以你可以把 Linux 中的执行实体近似理解成这样一个组合：

  • task_struct：执行者
  • mm_struct：地址空间
  • files_struct：打开文件集合

这三个东西拼起来，才更像一个完整的“执行上下文”。

2.3.1 mm_struct 里最值得展开的是什么

1. pgd

它指向页表顶级目录，是虚拟地址到物理地址映射的起点。

2. start_code / end_code

代码段边界。

3. start_data / end_data

数据段边界。

4. start_brk / brk

堆区边界，常和 brk / sbrk 相关。

5. start_stack

用户栈起始位置。

6. mmap

它指向 VMA 链表。VMA 可以理解成：

一段具有一致权限和用途的虚拟内存区域。

比如：

  • 代码段
  • 数据段
  • 堆
  • 栈
  • 动态库映射区
  • mmap 映射区

很多面试里问“虚拟内存怎么组织”，最后都绕不开 mm_struct 和 VMA。

2.4 进程状态不是背词条，而是理解“它现在在等什么”

面试时经常会背几个状态：

  • TASK_RUNNING
  • TASK_INTERRUPTIBLE
  • TASK_UNINTERRUPTIBLE
  • TASK_ZOMBIE
  • TASK_STOPPED
  • TASK_TRACED

但真正有用的理解方式不是死记，而是问自己：

这个任务现在是在跑、在睡、在等，还是已经结束但还没被回收？

2.4.1 重点看几个常见状态

TASK_RUNNING

不是说“正在占用 CPU”，而是表示：

  • 要么正在运行
  • 要么已经就绪，等待被调度

TASK_INTERRUPTIBLE

表示任务在睡眠，但可以被信号唤醒。

这类状态常见于：

  • 某些可中断等待
  • 定时等待
  • 普通阻塞 I/O 场景

TASK_UNINTERRUPTIBLE

表示不可中断睡眠，通常记成 D 状态。

这类状态常常和底层 I/O 等待、磁盘等待、某些设备等待有关。它之所以麻烦，是因为即使发 kill -9，也未必能立即结束。

根本原因不是“kill 失效”，而是：

任务当前处在一个不能安全响应信号的内核等待点。

TASK_ZOMBIE

说明任务已经退出，但退出信息还没被父进程回收。

这类进程不再执行，只保留最小信息，等父进程 wait。

TASK_STOPPED

通常表示被 SIGSTOP、SIGTSTP 之类信号暂停。

TASK_TRACED

通常表示进程正处于调试器跟踪状态，比如 gdb 断点暂停。

三、文件描述符：Linux I/O 抽象的入口

接下来要把进程 / 线程和 I/O 串起来，就必须看文件描述符。

Linux 最经典的一句抽象是：

一切 I/O 都可以通过 fd 进入内核。

这句话不是说所有东西都真的“长得一样”，而是说：

用户态统一通过文件描述符索引内核中的 file 对象，再由 file_operations 把不同类型对象的行为分发下去。

3.1 文件描述符的三层关系

flowchart LR
    A[用户态 fd] --> B[fdtable]
    B --> C[struct file]
    C --> D1[inode / 文件系统]
    C --> D2[socket / sock]
    C --> D3[pipe / 设备对象]

从用户到内核，大致是这样：

  1. 进程文件描述符表：fd 数组
  2. 内核 file 结构：打开文件的状态对象
  3. 更底层的对象：inode、socket、pipe、设备等

可以画成一个简化图：

也就是说：

  • fd 只是一个整数索引
  • file* 才是内核中的打开对象
  • 再往下，才是真实文件、socket 或设备

3.1.1 为什么 fd 只是“下标”，不是“对象本身”

因为在 Linux 里：

  • 一个进程可以把同一个 file* 映射到多个 fd
  • 父子进程 fork 后也可能持有指向同一个 file* 的不同 fd 表项
  • 真实状态，比如文件偏移、打开标志，更接近保存在 file 层，而不是 fd 整数本身里

所以你面试里如果只把 fd 理解成“文件句柄”，其实还不够。

更准确地说：

fd 是用户态访问内核 I/O 对象的一层索引。

关键就在：

  • 磁盘文件系统

四、I/O 的根本问题，不是“读不读”，而是“等不等”

很多人一开始学 I/O，容易盯着“数据怎么拷贝”，但在高并发场景里，真正更本质的问题通常是：

如果数据还没准备好，线程怎么办？

这才是阻塞、非阻塞、多路复用这些模型的起点。

4.1 阻塞 I/O

flowchart TD
    A[用户调用 read] --> B[进入内核]
    B --> C{数据是否准备好}
    C -- 否 --> D[线程睡眠]
    D --> E[数据到达后被唤醒]
    E --> F[内核拷贝数据到用户缓冲区]
    F --> G[返回用户态]
    C -- 是 --> F

阻塞 I/O 的流程通常是：

  1. 用户态发起系统调用
  2. 进入内核态
  3. 发现数据没准备好
  4. 线程睡眠
  5. 数据准备好后被唤醒
  6. 内核把数据拷回用户态
  7. 系统调用返回

好处是简单。
问题是，一个线程往往只能“等一个地方”。

4.2 非阻塞 I/O

flowchart TD
    A[用户调用 read] --> B[进入内核]
    B --> C{数据是否准备好}
    C -- 否 --> D[立即返回 EAGAIN]
    D --> E[用户继续轮询]
    C -- 是 --> F[内核拷贝数据]
    F --> G[返回用户态]

非阻塞不是“不进内核”，而是：

  • 如果没数据，不睡眠
  • 直接返回 -1 和 EAGAIN

  • 用户态切内核态
  • 查 fd
  • 查 file
  • 查对象状态
  • 再返回用户态

这会带来大量系统调用开销和 CPU 空转。

4.3 I/O 多路复用为什么会出现

flowchart LR
    A[用户关注多个 fd] --> B[把等待交给内核]
    B --> C[内核统一管理等待]
    C --> D[只有 fd 就绪时才唤醒线程]
    D --> E[用户处理 ready fd]

自然就会有一个更合理的需求：

能不能让内核帮我等很多 fd，但只在真正有事件时再唤醒我？

这就是 select / poll / epoll 出现的根本原因。

它们解决的核心，不是“让数据更快到达”，而是：

把等待这件事，从用户态低效轮询，转成内核态统一管理。

五、从网卡到 socket：数据到底怎么走进来

flowchart TD
    A[网卡收到数据包] --> B[DMA 写入 Ring Buffer]
    B --> C[触发硬中断]
    C --> D[驱动处理中断上半部]
    D --> E[软中断 / NAPI poll]
    E --> F[批量取包并构造 sk_buff]
    F --> G[协议栈逐层处理]
    G --> H[根据四元组找到 socket]
    H --> I[放入 socket 接收队列]
    I --> J[socket 变为可读]
    J --> K[回调通知 epoll]
    K --> L[epitem 进入 ready list]
    L --> M[唤醒 epoll_wait]

理解 epoll 之前，最好先看一眼数据路径。

当网卡收到数据包时，典型路径并不是“直接塞到用户程序里”，而是：

  1. 网卡通过 DMA 把数据写到内存缓冲区
  2. 触发硬中断通知 CPU
  3. 驱动快速处理中断上半部
  4. 交给软中断 / NAPI 继续处理
  5. 批量从 ring buffer 取包
  6. 封装成 sk_buff
  7. 交给协议栈逐层处理
  8. 根据四元组找到目标 socket
  9. 放入 socket 接收队列
  10. 标记 socket 变为可读
  11. 唤醒等待该事件的线程或 epoll 实例

5.1 为什么不能每个包都靠硬中断

如果高并发下每来一个包就触发一次硬中断，CPU 会很快被中断风暴打爆。

所以 Linux 用 NAPI 做平衡：

  • 第一个包来，先触发一次中断
  • 暂时关闭该队列中断
  • 改成软中断轮询，批量收包
  • 流量下去后，再重新开启中断

你可以把它理解成：

流量小时靠中断及时响应，流量大时靠轮询批量处理。

5.1.1 网卡 ring buffer 是什么

常见抽象可以写成：

驱动会组织出一个固定大小、连续内存、环形使用的描述符数组。网卡通过 DMA 往这些描述符指向的缓冲区写数据，内核再从中批量取包。

六、epoll 为什么快，本质不是“黑魔法”，而是少做无效扫描

很多人背结论时会记：

  • select / poll 是 O(N)
  • epoll 是 O(1)

但这里的核心不能只背复杂度，而是要知道“省掉了什么”。

6.1 select / poll 的痛点

用户态每次调用时，内核都要扫描整批关注的 fd，看谁准备好了。

也就是说，它更像：

  • 我每次都把名单重新交给你
  • 你每次都重新检查一遍

如果连接很多，但真正就绪的只有少数，这就很亏。

6.2 epoll 的核心思想

flowchart TD
    A[eventpoll] --> B[rbr 红黑树 关注列表]
    A --> C[rdlist 就绪链表]
    A --> D[wq 等待队列]
    B --> E[epitem]
    C --> E

epoll 把“关注谁”和“谁已经就绪”拆开管理：

  • 一份关注列表
  • 一份就绪列表

ready list 里现在有哪些就绪事件

这就是它快的根本原因。

6.3 epoll_ctl 和 epoll_wait 分别做什么

epoll_create1()

创建一个 eventpoll 实例。

epoll_ctl()

把 fd 注册进去，内核会为它创建一个 epitem 节点，并插入关注红黑树。

epoll_wait()

等待就绪事件，如果 ready list 非空，就直接取出返回；如果为空，就把当前任务挂到等待队列里睡眠。

七、为什么 epoll 能做到“fd 一就绪就被放进 ready list”

真正关键的一点在于：

epoll 不是靠 epoll_wait 时临时去扫全部 fd，而是靠回调机制在状态变化时把 fd 挂进 ready list。

当某个 socket 状态发生变化，比如变为可读时：

  1. 底层 socket 唤醒逻辑触发
  2. 调到 epoll 注册时挂上的回调
  3. 回调把对应 epitem 放进 rdlist
  4. 唤醒在 epoll_wait 上睡眠的任务

这就是 epoll 最有代表性的设计点：

事件驱动地维护就绪列表，而不是每次全量扫描。

八、再把整体串一遍

到这里，其实可以把整条线再收一次。

8.1 进程 / 线程层

  • Linux 调度的是 task_struct
  • 线程和进程的本质区别在资源共享
  • mm_struct 是区分地址空间是否共享的关键

8.2 I/O 抽象层

  • 用户态通过 fd 进入内核
  • fd 先找到 file*
  • file_operations 再把操作分发给具体对象

8.3 等待机制层

  • 阻塞 I/O：线程自己睡
  • 非阻塞 I/O：线程不断问
  • I/O 多路复用：让内核帮你等很多 fd

8.4 epoll 层

  • 红黑树管关注列表
  • ready list 管就绪列表
  • 回调在状态变化时把 fd 放进 ready list
  • epoll_wait 只取已经 ready 的事件

这样一来，进程、线程、文件、socket、epoll 其实并不是几块孤立知识，而是一整套连续设计。

九、总结

如果把本文压缩成几句话，就是：

  1. Linux 内核真正调度的是 task，而不是课本式抽象的“进程概念”。
  2. 进程和线程的核心区别，本质是资源是否共享，尤其是地址空间是否共享。
  3. Linux I/O 的统一入口是 fd，而真正的对象分发依靠 file 和 file_operations。
  4. I/O 模型的核心问题不是“读不读”，而是“数据没准备好时怎么等”。
  5. epoll 快，不是因为神秘，而是因为它把关注和就绪分开，并且避免了无效扫描。

如果你后面准备继续深入，我建议下一步顺着两条线走：

  • 一条是 fork / 写时复制 / 地址空间 / mmap / 栈堆模型
  • 一条是 epoll 深水区：LT/ET、惊群、回调、锁与等待队列

这样你的 Linux 体系会越来越完整，而不是停留在分散记忆。
