从分块到 NALU，一文讲清 H.264/H.265 的编解码流程与码流结构

上一篇我先把 H.264/H.265 的基础世界观搭起来了，重点解决的是：

  • 为什么视频一定要压缩
  • H.264 和 H.265 是什么
  • YUV、GOP、I/P/B 帧、码率这些概念该怎么理解

但如果只停在这一层，很多东西其实还是会悬着。

因为真正一进入工程或面试，最容易被问到的往往不是“它是什么”，而是：

  • 编码器到底做了哪些事？
  • 为什么要分块？
  • 什么叫帧内预测、帧间预测？
  • 量化为什么会丢信息？
  • NALU、SPS、PPS 到底是什么？
  • Annex B 和 AVCC 又是什么关系？

所以这一篇我就不再停留在“概念认识”，而是往真正更核心的一层走：

把 H.264/H.265 的编解码流程和码流结构串起来。

你可以把这一篇理解成：

从“视频是怎么被压缩的”，一路讲到“压缩结果最终长什么样”。

一、为什么“懂流程”比“背概念”更重要

很多人学视频编码时，容易陷入一种很典型的状态：

  • 知道 I 帧、P 帧、B 帧
  • 知道 SPS、PPS
  • 也听过运动估计、量化、CABAC

但这些词在脑子里是散的。

一旦被问到：

你从输入一帧图像开始，讲讲编码器到底做了什么。

就会立刻卡住。

原因很简单：

视频编码不是一堆孤立名词，而是一条完整流水线。

如果你只背词，不理解这些模块在流水线里的位置，就会出现两个问题：

  1. 你知道术语，但没有结构感
  2. 你很难把 H.264/H.265 和 FFmpeg、硬编硬解、推流拉流这些工程问题真正接起来

所以这一篇最重要的目标不是“记更多概念”，而是：

把编码器和解码器的整体流程看成一张图。

二、先看整体：编码器到底在做什么

可以先把编码器的总任务压缩成一句话：

尽量减少视频里的冗余信息，把原始帧变成更小的压缩码流。

如果拆成更具体的步骤，一条典型的视频编码主线大致是：

  1. 输入原始图像帧
  2. 对图像做分块
  3. 做帧内预测或帧间预测
  4. 对残差做变换
  5. 对变换结果做量化
  6. 做熵编码
  7. 输出压缩码流

这条链看起来不长，但里面每一步都在回答一个非常实际的问题：

  • 怎么找冗余
  • 怎么表示差异
  • 怎么把差异再压小
  • 怎么组织成可传输、可解码的码流

可以先看一个抽象流程图：

flowchart TD
    A[原始视频帧] --> B[分块]
    B --> C[帧内 / 帧间预测]
    C --> D[得到残差]
    D --> E[变换]
    E --> F[量化]
    F --> G[熵编码]
    G --> H[压缩码流输出]

后面整篇基本都在讲这张图里的各个模块。

三、分块机制：为什么编码器不按整帧硬压

如果直接把一整帧图像当成一个整体去处理，计算和建模都会非常困难。

所以视频编码首先会做一件很自然的事：

把图像分成更小的块，再逐块处理。

3.1 H.264 的宏块（Macroblock）

在 H.264 里，一个非常经典的单位就是宏块（Macroblock）。

你可以先粗略记住：

  • 宏块常见大小是 16×16
  • 编码器会围绕这个粒度做预测、变换、编码

当然，H.264 里也不是永远固定只按 16×16 不动，它也有更细粒度的子分块思想，但在入门理解阶段，先把“宏块是主要基本处理单元”抓住就够了。

3.2 H.265 的 CTU / CU / PU / TU

到了 H.265，分块设计变得更灵活。

它不再主要强调传统 H.264 那种“宏块”概念，而是引入了更复杂的层次结构，比如：

  • CTU（Coding Tree Unit）
  • CU（Coding Unit）
  • PU（Prediction Unit）
  • TU（Transform Unit）

如果第一次看这些名字会晕，先别急着全背。

更重要的理解是：

H.265 允许编码器按更灵活、更自适应的方式去切图像块。

这件事为什么重要？

因为真实图像不是规则棋盘。

  • 平坦区域适合大块表示
  • 边缘复杂区域适合细分成小块

H.265 正是通过这种更灵活的块划分，进一步提高压缩效率。

3.3 为什么 H.265 压缩效率更高，和分块机制有很大关系

这个问题非常值得记。

H.265 更省码率，不只是因为“算法更先进”这么抽象，背后很重要的一层就是：

它在分块和预测上比 H.264 更灵活，更能贴近图像真实结构。

所以你后面如果面试被问：

H.264 和 H.265 核心差异是什么？

除了说压缩率更高，你还可以补一句：

  • H.265 的块划分机制更灵活
  • 更适合处理复杂场景和高分辨率画面

这会显得你不是只会背结论。

四、帧内预测：如何利用空间冗余

前一篇提到过，视频里有空间冗余，也就是：

同一帧中，相邻像素往往相似。

帧内预测就是在利用这个特点。

4.1 帧内预测在做什么

可以用一句很直白的话来概括：

我不直接把这块像素原封不动写出去，而是先猜它大概长什么样，再只记录“猜得不准的差异”。

也就是说，编码器会根据：

  • 左边的像素
  • 上边的像素
  • 邻近已经编码好的区域

来预测当前块大概应该长什么样。

然后真正要编码的，不再是完整像素值，而是：

实际值 - 预测值 = 残差

如果预测得好，这个残差就会比较小，后面压起来就更容易。

4.2 为什么这件事有效

因为自然图像里，相邻区域通常高度相关。

比如：

  • 一块天空
  • 一块墙面
  • 一张脸的大面积平滑区域

这些地方像素变化通常不会太剧烈。

所以用周围区域去猜当前块，很多时候是有效的。

4.3 可以怎样直觉理解“预测模式”

你不用一上来就背所有模式，只要先理解：

  • 有些模式偏“水平延展”
  • 有些模式偏“垂直延展”
  • 有些模式偏“斜向延展”
  • 有些模式会做更复杂的平滑估计

本质就是：

编码器在尝试不同方向和方式，找到最适合当前块纹理结构的预测方式。

五、帧间预测：如何利用时间冗余

如果说帧内预测是在同一帧里找相似性，那么帧间预测就是在：

不同帧之间找相似性。

这也是视频压缩里最有代表性的一部分。

5.1 视频为什么特别适合做帧间预测

因为视频不是一堆毫无关系的静态图片。

前后两帧往往差别很小。

比如一个人在镜头前说话：

  • 背景基本不动
  • 身体大部分区域位置接近
  • 真正变化明显的，可能只是嘴、眼、手势

所以编码器没必要每一帧都从头开始完整编码。

5.2 运动估计和运动补偿

这两个词一定会经常看到。

运动估计（Motion Estimation）

编码器会尝试找：

当前块在参考帧里最像哪一块？

比如当前块看起来像“上一帧左上角偏移了几个像素后的位置”。

运动补偿（Motion Compensation）

找到这个“最像的块”之后，就可以用它来作为预测结果。

剩下再编码真实块和预测块之间的差异。

所以你可以把帧间预测简单理解成：

  • 先去前后帧里找相似块
  • 再用位移 + 残差来表示当前块

5.3 为什么这是视频压缩里最值钱的一步

因为时间冗余通常非常大。

尤其在：

  • 监控场景
  • 固定机位场景
  • 直播场景
  • 慢运动场景

如果帧间预测做好，压缩效率会非常可观。

但反过来，它也会带来更大的计算复杂度，因为“找相似块”本身并不轻松。

六、残差、变换与量化：为什么编码器不是直接存“差异值”

前面不管是帧内预测还是帧间预测，都会产生一个东西：

残差（residual）。

也就是：

如果预测足够准，这个残差会更“稀疏”、更容易压缩。

但编码器还不会直接把残差原样写出去，而是通常还要再走两步：

  • 变换
  • 量化

6.1 为什么要做变换

变换的目标不是“凭空减少信息”，而是：

把原本在空间域里的残差，转换到更适合压缩的表示形式。

通俗一点说，就是把“像素差异”换一种表达方式，让它更容易出现：

  • 大量接近 0 的值
  • 更集中的能量分布

这样后面量化和熵编码就更容易奏效。

6.2 为什么量化会带来有损压缩

量化这一步非常关键，因为：

真正让视频编码变成“有损”的，量化通常是核心来源之一。

量化的本质就是：

  • 对某些值做更粗粒度处理
  • 丢掉一部分细节精度
  • 换取更高压缩率

这时候就开始进入经典 trade-off：

  • 量化更狠 → 文件更小，画质更差
  • 量化更轻 → 文件更大，画质更好

6.3 QP 为什么重要

你以后一定会反复看到 QP（Quantization Parameter）。

因为它基本上是在控制：

压缩得有多狠。

它和下面这些都高度相关：

  • 码率
  • 清晰度
  • 失真程度
  • 主观观感

所以很多编码参数调优，最后都会绕回到量化策略和 QP。

七、熵编码：最后怎么把数据真正压成码流

到了前面这一步，编码器手上已经拿到了经过预测、变换、量化之后的数据。

接下来就要进入最后一层：

把这些中间结果进一步组织成真正的压缩比特流。

这一步常常叫：

  • 熵编码

7.1 熵编码在做什么

它不负责理解图像语义，也不负责预测。

它更像是在做：

如何用更短的比特表示更常见的信息，用更长的比特表示更少见的信息。

所以它本质上是统计意义上的压缩。

7.2 H.264 里的 CAVLC 和 CABAC

H.264 里常会提到两种熵编码方式：

  • CAVLC
  • CABAC

入门阶段先抓这个区别就够了：

  • CAVLC 更简单
  • CABAC 压缩效率通常更高，但复杂度也更高

7.3 H.265 为什么主要强调 CABAC

H.265 更偏向用 CABAC 这一类更高效的熵编码手段。

所以它能进一步压榨码率，但也带来更高实现复杂度。

八、环路滤波：为什么编码器内部还要“回头修图”

很多人第一次看到环路滤波会疑惑：

不是已经编码完了吗，为什么还要再处理一次？

因为编码器内部并不是只关心“写出当前帧”，它还关心：

后续帧会不会拿当前帧做参考。

8.1 为什么会有块效应

由于分块编码和量化，图像边界上可能会出现比较明显的块状痕迹。

这时候如果直接把这种“粗糙重建结果”拿去当参考帧，后续预测效果也会受影响。

8.2 去块滤波和 SAO

所以编码器通常会在重建图像上做一些后处理，比如：

  • 去块滤波（Deblocking Filter）
  • SAO（Sample Adaptive Offset，H.265 中更常见）

本质目的都是：

  • 改善重建图像质量
  • 让后续参考帧更稳定

所以这一步不是“可有可无的美颜”，而是整个编码闭环的一部分。

九、解码流程：为什么说它大体是编码的逆过程

如果把编码看成“压缩”，那解码就是“重建”。

解码器要做的事，大体是把码流重新还原成图像。

可以粗略看成下面这条线：

  1. 熵解码
  2. 反量化
  3. 反变换
  4. 重建预测结果
  5. 得到图像帧
  6. 维护参考帧

可以画成一个更直观的流程：

flowchart TD
    A[压缩码流] --> B[熵解码]
    B --> C[反量化]
    C --> D[反变换]
    D --> E[重建预测块]
    E --> F[输出图像帧]
    F --> G[作为后续参考帧]

9.1 为什么说“是逆过程”，但又不能简单理解成完全镜像

因为整体方向上它确实是逆过程：

  • 编码时做了什么压缩动作
  • 解码时就要做相应还原动作

但它也不是数学上逐项完全“反着走一遍”那么简单。

因为：

  • 量化已经丢了信息
  • 解码器恢复的是“重建图像”，不是原始无损图像

所以你更应该把它理解成：

大体按编码的反方向重建，而不是恢复绝对原始值。

9.2 为什么解码器也要维护参考帧

因为 P 帧、B 帧解码时同样需要参考别的帧。

所以解码器不是“解完就忘”，它也必须维护一套参考帧管理机制，否则后面的帧就没法正确恢复。

十、码流结构：压缩结果最终长什么样

前面讲的是“编码器内部怎么做压缩”，接下来要回答另一件同样重要的事：

压出来的数据，最后到底长什么样？

这就进入码流结构层。

10.1 NALU 是什么

NALU 全称可以理解成网络抽象层单元（Network Abstraction Layer Unit）。

你可以先把它理解成：

视频码流中的基本组织单元之一。

也就是说，压缩结果不是一整坨没有边界的二进制，而是会被切分成一个个有类型、有边界意义的数据单元。

这点非常重要，因为后面无论：

  • 存文件
  • 做封装
  • 推流
  • 解析参数集

都会反复碰到 NALU。

10.1.1 为什么要有 NAL 这一层

因为编码器内部产生的是“压缩后信息”，但这些信息还要被：

  • 传输
  • 存储
  • 解封装
  • 交给解码器识别

所以需要一种统一组织方式，把不同类型的信息块区分开。

NALU 就是在做这件事。

10.2 SPS / PPS / VPS 是什么

如果说 NALU 是大框架，那么参数集就是其中最关键的一类内容。

10.2.1 SPS

SPS 可以理解成序列级参数集。

里面通常会放一些对整个视频序列都很关键的解码信息，比如：

  • 分辨率相关信息
  • 编码级别相关信息
  • 一些全局解码参数

10.2.2 PPS

PPS 可以理解成图像级参数集或者更细粒度控制参数集。

它会补充一些更具体的编码控制信息。

10.2.3 VPS

VPS 是 H.265 里更常出现的一层，可以理解成更高层的视频参数集。

10.2.4 为什么参数集这么重要

因为解码器如果不知道这些参数，就很难正确解释后面的码流。

所以很多花屏、解码失败、播放器打不开的问题，最后都可能和参数集缺失或不一致有关。

10.3 Annex B 和 AVCC 是什么关系

这是工程里非常常见、也非常容易混的一个点。

10.3.1 Annex B

Annex B 风格的码流，常见特征是：

用起始码（start code）来分隔 NALU。

比如你会看到典型的：

它们就是 NALU 的边界标记。

10.3.2 AVCC

AVCC 这一类组织方式，更常见的特点是：

不用起始码，而是用长度字段来描述后面这个 NALU 有多长。

10.3.3 为什么要区分这两种

因为不同封装格式、播放器、解封装器、FFmpeg 模块，对输入码流的期望可能不同。

有时候你以为“都是 H.264”，但其实问题不是编码标准不一样，而是：

码流组织格式不一样。

这也是为什么工程里经常会听到：

  • Annex B 转 AVCC
  • AVCC 转 Annex B

10.4 裸流和封装格式不要混

这是初学者特别容易踩的坑。

你要明确区分两层：

1. 码流层

比如：

  • 裸 H.264
  • 裸 H.265

这一层关心的是：

  • NALU 怎么组织
  • 参数集在哪里
  • 码流边界怎么表示

2. 容器 / 封装层

比如：

  • MP4
  • FLV
  • TS
  • MKV

这一层关心的是：

  • 音视频流怎么一起装
  • 时间戳怎么管理
  • 元数据怎么组织
  • 文件格式怎么定义

所以不要把：

  • H.264
  • MP4

当成同一层概念。

更准确地说：

H.264/H.265 是编码标准，MP4/TS/FLV 是封装格式。

十一、H.264 和 H.265 在流程上的核心差异再收一次

到了这里，可以把两者差异再从“流程和结构”角度收一下。

11.1 H.265 的主要优势

  • 块划分更灵活
  • 预测机制更复杂
  • 压缩效率更高
  • 高分辨率场景更有优势

11.2 H.265 的代价

  • 编码更重
  • 解码更复杂
  • 工程落地成本更高
  • 终端支持要求更高

所以 H.265 的优势不是无代价的。

这也解释了为什么今天两者依然并存。

十二、总结

如果把这一篇压缩成一句话，那就是：

H.264/H.265 的核心，不只是“把视频压小”，而是通过分块、预测、变换、量化、熵编码和码流组织，构建出一套既能高效压缩、又能稳定解码的完整体系。

这一篇最重要的几个点，可以再收成下面这几句：

  1. 编码器的核心工作，是先找冗余，再压缩差异。
  2. 帧内预测利用空间冗余，帧间预测利用时间冗余。
  3. 变换和量化让压缩真正变得高效，而量化也是有损的关键来源。
  4. NALU、SPS、PPS、VPS、Annex B、AVCC 这些内容，决定了码流最终怎么被组织和识别。
  5. H.265 更高效，但也更复杂；H.264 更成熟，也更稳。
