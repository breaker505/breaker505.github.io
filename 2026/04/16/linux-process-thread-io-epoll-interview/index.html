Linux 进程、线程、文件与 epoll 高频面试题整理

整理 Linux 时，一个常见问题是：

知识点和面试题很容易混在一起。

知识点文章更适合搭体系，而面试题更适合做“查缺补漏”和“高频追问准备”。所以这篇单独把进程、线程、文件描述符、I/O 模型和 epoll 相关的高频问题拎出来，按主题做一版更像面试复盘的整理。

说明：下面部分题目给的是直接面试回答思路，部分深水区问题先给关键结论和理解方向，方便后续继续展开。

一、进程 / 线程 / 文件描述符高频题

1. 如果两个进程通过 fork 产生，它们的 fd 表是共享的，还是复制的？

一个更完整、也更像面试的回答方式是：

fd 表本身会复制，但很多 fd 表项最终指向的 struct file 对象是共享的。

也就是说：

  • 父子进程各自有一份 fdtable
  • 但这些 fdtable 里的很多条目会指向同一个 file*

所以进一步会带来一个结果：

  • 文件偏移可能共享
  • 打开状态可能共享
  • 某些 file 层面的标志可能共享

如果面试官继续追问“为什么父子进程写同一个 fd 会互相影响”，你就可以顺着这个点继续展开。

2. 非阻塞 + 轮询的复杂度是 O(N)，这个 N 是什么？

这里的 N 指的是：

你当前维护并且需要轮询检查的 fd 总数。

可以直接说得更白一点：

不是当前 ready 的数量，而是你维护的连接总数。

所以当连接很多但真正活跃的很少时，非阻塞 + 用户态轮询会特别浪费。

3. 堆和栈有什么区别？

建议别背成死板表格，而是按“管理方式 + 空间特征 + 生命周期 + 常见问题”来讲。

一个自然版回答可以是：

  • 栈主要由编译器和运行时自动管理，函数调用时分配，返回时自动回收
  • 堆主要由程序主动申请和释放，比如 malloc/free、new/delete
  • 栈通常是连续空间，访问快，但空间相对有限
  • 堆更灵活，适合动态分配大对象，但可能带来碎片和泄漏问题

如果面试官再追问“为什么栈快”，你可以继续补：

  • 分配回收通常只是移动栈指针
  • 不像堆需要复杂分配器管理

4. mmap 在哪里？

常见回答：

在用户态虚拟地址空间里，通常位于堆和栈之间的映射区域。

如果要更稳一点，可以补一句：

  • 具体布局和体系结构、内核实现有关
  • 但一般理解成“堆和栈之间的大块映射区”没有问题

5. 线程栈怎么分配？

建议分开说：

  • 每个线程通常会有独立的用户栈
  • 每个 task 也会有自己的内核栈

也就是说，线程不是“完全共享栈”，而是：

  • 共享地址空间
  • 但各自有独立执行栈

6. 栈溢出怎么产生？

本质上是：

线程在使用栈时，越过了分配给它的栈边界。

常见原因：

  • 递归太深
  • 局部变量太大
  • 死递归
  • 线程栈本来就配置得比较小

7. new / malloc 底层一定走 brk 吗？

不一定。

一个更像面试回答的说法是：

  • 小块分配常常更可能走 brk
  • 大块分配常常更可能走 mmap
  • 具体阈值和行为依赖分配器实现

所以不要把它回答成“只走 brk”或者“只走 mmap”。

8. fork 为什么高效？

这是一个很经典的问题。

可以这样答：

fork 高效的核心原因在于：

  • 它不会一上来就把父进程的全部物理内存拷一份给子进程
  • 更常见的做法是复制必要内核结构和页表
  • 物理页先共享
  • 真正写的时候再触发 COW

所以它高效，不是因为“复制很快”，而是因为：

大量内存复制被延后到了真正写入时。

9. fork 底层原理怎么说？

建议抓这三步：

  1. 创建新的 task 结构
  2. 复制页表，但不立刻复制物理页
  3. 设置写时复制

如果想答得更像体系理解，可以说：

  • 复制 task_struct 等核心内核结构
  • 为子进程建立自己的页表视图
  • 父子共享物理页并设为只读
  • 一旦写入再分裂物理页

10. fork 后父子谁先执行？

不确定。

由调度器决定，不能写死。

这题不要掉进“父一定先”或者“子一定先”的坑。

11. fork 两次打印几次？

如果没有条件控制，通常是 4 次。

因为：

  • 第一次 fork 后有 2 条执行流
  • 第二次再各自 fork，一共变成 4 条

12. fork 在多线程程序里安全吗？

这题答的时候要体现“谨慎”。

可以说：

  • 多线程程序里 fork 后只保留调用线程
  • 其他线程消失
  • 但很多锁状态、运行库状态可能留在一种不完全一致的状态
  • 所以工程上更常建议 fork 后尽快 exec

13. 写时复制怎么实现？

本质流程：

  • 父子共享同一物理页
  • 页表先改成只读
  • 某一方写入时触发缺页异常
  • 内核分配新页并复制内容
  • 更新页表，让写入方指向新页

14. 父子进程共享什么？

这个问题不要答成绝对化。

更好的说法是：

  • 初始阶段可能共享物理页（通过 COW）
  • 可能共享同一个 file 对象
  • 会共享代码逻辑
  • 但它们有各自独立的进程语义和地址空间管理视图

15. D 状态是什么？为什么 kill -9 杀不掉？

D 状态通常对应 TASK_UNINTERRUPTIBLE。

典型说法是：

  • 它常见于某些底层 I/O 等待、设备等待、磁盘等待场景
  • 此时任务不处于能安全处理中断信号的点
  • 所以你发 kill -9，信号送到了，但任务暂时响应不了

16. 为什么线程切换比进程切换开销小？

常见理解是：

  • 线程切换通常共享地址空间
  • 进程切换通常要切换地址空间
  • 地址空间切换会涉及页表、TLB、缓存局部性等更多成本

所以线程切换通常更便宜，但注意也不要说成“线程切换几乎没成本”。

二、epoll 基础高频题

17. 三次握手发生在哪个函数？

建议别答成“发生在 listen()”。

更准确的回答是：

三次握手是在客户端调用 connect() 后，由内核 TCP 协议栈自动完成的。

  • listen() 只是让 socket 进入监听状态
  • connect() 才会触发客户端发 SYN
  • 内核协议栈自动完成后续握手
  • accept() 只是把已完成连接取出来交给应用层

18. 同一个 fd 被触发 5 次，ready list 里会有 5 份吗？

通常不会。

因为同一个 epitem 节点不会被重复插入 ready list。内核会通过标志位或状态控制避免重复入链。

19. 为什么 epoll 的关注列表用红黑树，不用哈希表？

可以从这几个点答：

  • 内核已有成熟实现
  • 插入 / 删除 / 查找有稳定复杂度
  • 有序结构利于管理
  • 避免额外哈希设计和退化问题

20. epoll 红黑树的 key 是什么？

更底层看，重点不是裸整数 fd，而是 内核对象语义。

常见理解是：

  • fd 是进程级索引
  • struct file* 才是内核级对象

所以面试里最好体现出“fd 只是入口，file 才是核心对象”。

21. 如果 key 是 file*，为什么 epoll_ctl 还要传 fd？

因为用户态只能持有 fd，内核需要先根据 fd 找到 file*，再把它纳入 epoll 管理结构。

22. 为什么不同 epoll 实例可以监听同一个 file*？

因为 epoll 并不独占底层对象。

更准确地说：

  • epoll 会把自己挂到目标对象的等待机制上
  • 一个底层对象可以对应多个等待者
  • 所以不同 epoll 实例可以同时监听它

23. epoll_ctl(ADD) / MOD / DEL 的复杂度是多少？

通常回答：

  • ADD：O(logN)
  • MOD：常按 O(logN) 记
  • DEL：O(logN)

重点在于：这些操作是针对关注集合，不是扫描全部 ready 事件。

24. ready list 为什么用链表，不用红黑树？

因为 ready list 的目标不是按 key 查找，而是：

快速顺序取出已经 ready 的节点。

25. ready list 为什么是双向链表？

因为可能需要 O(1) 地把某个节点从中间移除，例如删除监听时该节点可能已经在 ready list 里。

26. epitem 会不会被重复插入 ready list？

正常不会。

这是 epoll 设计里一个关键点，否则一个高频 ready fd 会把 ready list 冲爆。

27. epoll_event 在内核里存在哪里？

通常会保存在与该 fd 对应的 epitem 里。epoll_wait 返回时，再把这些事件信息拷回用户态。

28. epoll_wait 为什么近似 O(1)？

更准确的说法是：

事件分发阶段与总 fd 数解耦，只与 ready 数相关。

所以很多资料会把它近似叫 O(1)，本质是“不需要每次扫描全部关注 fd”。

三、epoll 触发模式与事件语义

29. LT 模式的本质是什么？

Level Trigger 的本质是：

只要条件还满足，就可以持续通知。

比如 socket 里数据没读完，下次 epoll_wait 仍然会看到它可读。

30. ET 模式的本质是什么？

Edge Trigger 的本质是：

更强调状态变化时刻，而不是持续状态本身。

所以它更高效，但也要求应用层更小心。

31. ET 为什么必须读到 EAGAIN？

因为你要主动把当前可读数据尽可能读干净，不然剩余数据还在，但新的边沿变化未必再出现。

32. ET 会不会丢事件？

严格说不是 epoll 自己“丢了”，而是应用层如果没按 ET 语义把数据处理干净，就可能错过后续处理时机。

33. LT 一定不会出错吗？

也不绝对。

LT 更宽容，但应用层仍可能因为并发、状态管理或逻辑错误出现问题。

34. epoll 保存的是“状态”还是“边沿”？

更稳的说法是：

  • epoll 依赖底层对象的状态变化来触发 ready
  • LT / ET 则决定了用户该如何理解和消费这些 ready 事件

四、锁、等待队列与惊群问题

35. wq 上挂的到底是什么？

挂的是等待 epoll_wait 的任务，也就是等待队列项对应的线程 / 进程。

36. 如果三个线程同时 epoll_wait，会怎样？

它们都会进入对应等待队列。真正唤醒几个、谁先抢到资源，要看唤醒策略、调度和是否用了 EPOLLEXCLUSIVE。

37. 什么是惊群问题？

一个事件到来，本来只需要一个线程处理，却唤醒了一堆等待线程，最后只有一个拿到机会，其他线程白白被调度起来。

38. EPOLLEXCLUSIVE 怎么缓解惊群？

它的目标就是减少一个 ready 事件把大量等待者全部唤醒的情况。

39. 多线程同时处理一个 ready fd 会怎样？

如果应用层没做好同步和状态控制，可能会出现：

  • 重复消费
  • 竞争读写
  • 一方已经读空另一方还在处理
  • ET 场景下更容易出问题

所以 epoll 负责通知，不负责帮你保证业务层并发正确。

五、完整内核路径类题

40. 从网卡中断开始，数据如何走到 epoll？

建议按这条线答：

  1. 网卡 DMA 写入 ring buffer
  2. 触发硬中断
  3. 驱动处理中断上半部
  4. 触发软中断 / NAPI poll
  5. 批量取包并构造 sk_buff
  6. 协议栈逐层处理
  7. 根据四元组找到目标 socket
  8. 数据进入 socket 接收队列
  9. socket 状态变为可读
  10. 回调把对应 epitem 放入 ready list
  11. 唤醒 epoll_wait 等待者

41. ep_poll_callback 做了什么？

核心回答就是：

  • 在底层 fd ready 时被调用
  • 把对应 epoll 节点放进 ready list
  • 必要时唤醒等待队列

42. 为什么回调里不能做重操作？

因为这条路径处在很敏感的事件通知环节，做太重会拖慢整个事件分发效率。

43. epoll_wait 返回前，数据怎么到用户态？

注意区分：

  • epoll_wait 只是返回 ready 事件
  • 真正业务数据还要靠后续 read / recv 从内核拷回用户态

这题很适合拿来区分“事件通知”和“数据读取”不是一回事。

六、系统设计类问题

44. 10 万连接应该用几个 epoll 实例？

不要死答数字。

更好的回答是：

  • 取决于单 Reactor / 多 Reactor 模型
  • 取决于机器核数
  • 取决于是否配合 SO_REUSEPORT
  • 取决于连接分片和业务线程设计

45. 多核机器为什么常配合 SO_REUSEPORT？

因为它可以让多个监听 socket 绑定同一端口，把连接更自然地分散给不同核上的处理单元，降低 accept 竞争。

46. 单 Reactor 和多 Reactor 区别？

  • 单 Reactor：一个事件循环处理 accept、读写、分发
  • 多 Reactor：主 Reactor 处理 accept，从 Reactor 处理读写和事件分发

47. 线程池和 epoll 的正确关系是什么？

epoll 更偏 I/O 事件通知，线程池更适合耗时业务处理。不要把重计算直接塞进事件循环线程里。

七、边界与深水区问题

48. fd ready 时被 close 会怎样？

这个问题本质和对象生命周期、引用计数、并发安全有关。

面试里可以先说：

  • 内核不会简单粗暴地立即释放一切
  • 会依赖引用计数和回收时机保证基本安全
  • 但应用层仍可能因为时序问题出 bug

49. fork 之后 epoll 会怎样？

这题不要装得太绝对。

可以先说：

  • 相关资源会通过继承机制进入子进程可见范围
  • 但父子进程后续如何共享和处理 epoll 事件要非常谨慎
  • 工程上通常不建议在复杂 epoll 模型里随意 fork

50. 为什么 regular file 通常不适合 epoll？

因为普通文件通常总是 ready 的，缺少 socket 那种“等待某个外部事件到来”的语义，所以 epoll 在这类对象上没有太大意义。

51. epoll 能完全替代 select / poll 吗？

不能简单这么说。

在高并发网络场景里 epoll 很强，但不同接口有不同使用边界，面试里不要答成绝对替代关系。

八、最后怎么准备这部分面试

这一块很容易陷入两个误区：

  • 只背结论，不会串路径
  • 只记源码术语，不会讲成人话

我更建议按三层准备：

第一层：基础题讲清楚

比如：

  • fork
  • COW
  • fd 三层结构
  • 阻塞 / 非阻塞
  • epoll 为什么快

第二层：完整路径讲顺

比如：

  • read(fd) 怎么找到对象
  • 数据从网卡到 epoll 的完整流程
  • epoll 为什么不用扫描所有 fd

第三层：深水区能兜住

比如：

  • LT / ET 区别
  • 惊群
  • EPOLLEXCLUSIVE
  • D 状态
  • 线程切换和进程切换开销差异

面试真正拉开差距的，往往不是你背了多少题，而是：

你能不能把这些问题讲成一张连贯的图。

总结

如果把这篇压缩成一句话，那就是：

Linux 面试里，进程、线程、文件描述符、I/O 模型和 epoll 不是五个孤立章节，而是一条从 task 到 file、从等待到事件分发的完整链路。

你真正需要准备的，不只是单题答案，而是把这些答案装进同一套系统理解里。
