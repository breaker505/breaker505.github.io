音视频系统到底在处理什么：从数据形态到工程链路的整体认知

前言：为什么很多人学了很久音视频，脑子里还是一团雾

很多人学音视频时都会经历一个阶段。

一开始接触到的是各种具体名词：

  • H.264
  • H.265
  • AAC
  • RTP
  • RTMP
  • WebRTC
  • FFmpeg
  • AVPacket
  • AVFrame
  • YUV
  • NV12
  • OpenGL

再往后，又会不断碰到各种具体现象：

  • 为什么视频这么大
  • 为什么要压缩
  • 为什么会卡顿
  • 为什么会花屏
  • 为什么会音画不同步
  • 为什么 packet 不等于 frame
  • 为什么明明解码了还不能直接显示

表面上看，你知道的名词越来越多；但如果脑子里始终缺一张“总图”，这些知识点就很容易变成碎片。

最后就会出现一种很典型的状态：

每个词都见过，但整条链路还是串不起来。

我觉得这不是因为你不够努力，而是因为很多音视频资料默认你已经知道：

音视频系统真正处理的，其实不是“视频”这两个字，而是不同形态的媒体数据。

这句话很关键。

因为只要把“数据形态”这条主线抓住，很多原本看起来很散的问题，就会突然变得很顺。

所以这篇文章的目标，就是做这件事：

把音视频系统里最核心的几种数据形态，和它们之间的转换关系讲清楚。

如果这件事立住了，后面再去看编码、封装、传输、FFmpeg、显示、弱网、Camera Pipeline，都会容易很多。

一、先给一句最重要的话：音视频系统本质上是在处理“媒体数据的多次形态转换”

如果让我先给一句话结论，我会这样说：

音视频系统本质上不是在处理单一的“视频文件”或“音频流”，而是在处理媒体数据在不同阶段的多次形态转换。

比如同一段视频，在系统里可能先后以这些形态存在：

  • 原始采样数据
  • 原始图像帧 / 原始音频采样帧
  • 压缩码流
  • 封装后的文件数据
  • 网络传输数据
  • 接收端解析后的压缩包
  • 解码后的原始帧
  • 最终渲染 / 播放输出的数据

你会发现，这里面根本不只是一种“视频”。

它其实在不断变化形态。

而音视频工程真正干的事，就是围绕这些形态做：

  • 采集
  • 处理
  • 压缩
  • 组织
  • 传输
  • 还原
  • 渲染
  • 播放

一旦你从“数据形态转换”这个角度去理解，很多模块的职责就会清楚很多。

二、先看总图：音视频系统里常见的数据形态有哪些

先看一张总图，我们把最常见的几个层次摆出来。

flowchart LR
    A[真实世界声音 / 图像] --> B[采样后的原始数据]
    B --> C[原始帧 Frame]
    C --> D[压缩码流 Bitstream]
    D --> E[封装数据 Container Data]
    E --> F[网络传输数据 Stream / Packet]
    F --> G[接收端解析后的压缩数据]
    G --> H[解码后的原始帧]
    H --> I[渲染 / 显示 / 播放输出]

注意，这张图不是说所有系统都严格经过每一层，而是说：

音视频系统最核心的几种数据形态，大体就活在这些层次里。

接下来我们就按这个顺序，一层层往下讲。

三、第一层：真实世界信号，系统最开始面对的不是“视频文件”，而是光和声

这是很多人最容易下意识忽略的一层。

因为软件工程师平时看见的，往往已经是文件、流、帧或者 buffer 了。

但如果从整个系统看，音视频最开始面对的其实是：

  • 光学场景
  • 声音波动

也就是：

  • Camera sensor 面对的是光
  • 麦克风面对的是声波

所以整个音视频系统最源头的一步，是：

把连续的现实世界信号，采样成数字数据。

这一步如果是视频，对应的是图像采样；如果是音频，对应的是声音采样。

这也是为什么音视频工程天然和“采样”这个概念绑得很深。

四、第二层：采样后的原始数据，为什么它通常不能直接拿来存和传

现实世界被采样之后，系统拿到的并不是“轻便好用的小文件”，而往往是非常原始的数据。

4.1 对视频来说

视频采样后拿到的，通常是：

  • Bayer raw
  • YUV 原始帧
  • RGB 原始图像

这些数据的特点是：

  • 信息量大
  • 体积大
  • 还没压缩
  • 通常更适合后处理，而不是直接分发

4.2 对音频来说

音频采样后常见的是：

  • PCM 数据

它的特点也类似：

  • 没压缩
  • 数据连续
  • 更接近“真实采样结果”

所以从这层开始，你就能理解一个特别重要的事实：

原始数据最接近真实世界，但它往往也最笨重。

这决定了后面必须出现：

  • 编码
  • 压缩
  • 封装
  • 传输优化

否则系统根本不现实。

五、第三层：原始帧 Frame，是音视频系统里最关键的一种中间形态

接下来进入一个非常核心的概念：

Frame。

5.1 视频里的 Frame 是什么

对视频来说，frame 可以理解成：

一帧完整的原始图像数据。

它可能是：

  • YUV420P
  • NV12
  • RGB
  • Bayer 处理后的图像帧

Frame 的特点是：

  • 还原度高
  • 方便做处理
  • 容易做缩放、裁剪、滤镜、推理、显示
  • 但体积通常比较大

5.2 音频里也有 frame 的概念

音频里虽然直觉上不像视频那样“有一帧画面”，但工程上同样也有 frame 的概念。

比如一小段连续的 PCM 采样数据，也可以被组织成音频 frame 来处理。

5.3 为什么 Frame 这么重要

因为很多工程处理逻辑，本质上都发生在 frame 层：

  • 图像处理
  • 色彩转换
  • 缩放
  • 旋转
  • OSD 叠加
  • AI 前处理
  • 视频渲染
  • 音频特效 / 重采样

也就是说：

frame 更接近“内容本身”，而不是传输形式。

这一点后面理解 packet、bitstream、filter、显示时会非常有帮助。

六、第四层：压缩码流 Bitstream，为什么它一出现，整个世界就变了

一旦系统不只是想“处理内容”，而是还想：

  • 存储
  • 节省带宽
  • 网络分发
  • 长时间录像
  • 跨设备传输

那就必须进入压缩阶段。

压缩之后，数据形态会发生一个特别大的变化：

从原始 frame，变成压缩码流 bitstream。

6.1 视频里的 bitstream 常见是什么

比如：

  • H.264 码流
  • H.265 码流

6.2 音频里的 bitstream 常见是什么

比如：

  • AAC 码流
  • Opus 码流

6.3 为什么码流层这么重要

因为从这一层开始，数据不再主要为“直接内容处理”服务，而是更偏向：

  • 高效存储
  • 高效传输
  • 标准化分发

同时，这也意味着：

  • 你不能直接在码流层随便做图像处理
  • 你看到的不再是直观像素，而是压缩后的结果
  • 系统问题会开始涉及码率、GOP、时间戳、参数集、兼容性

换句话说，frame 层更像内容层，bitstream 层更像压缩表达层。

这句话值得反复记住。

七、第五层：封装数据，为什么“码流”和“文件格式”不是一回事

很多刚学音视频的人，在这里最容易混。

比如会说：

  • MP4 是一种编码格式
  • H.264 是一种文件格式

这两个说法都不对。

7.1 码流和容器分别是什么

  • H.264 / H.265 / AAC 这种，是编码后的码流格式
  • MP4 / FLV / TS / MKV 这种，是容器 / 封装格式

容器这层做的事，本质上是：

把压缩后的音视频数据，按照一定规则组织起来。

它解决的问题包括：

  • 音频流和视频流怎么放在一起
  • 时间信息怎么记录
  • 索引怎么组织
  • 文件结构怎么安排

7.2 为什么封装层很关键

因为只靠码流本身，很多工程问题是不方便解决的。

比如：

  • 播放时怎么定位
  • 多路媒体怎么同步
  • 文件怎么管理

所以你可以把封装层理解成：

给压缩媒体数据加上一层更适合存储和分发的组织方式。

八、第六层：网络传输数据，为什么同样一段视频，一上网络又是另一个世界

如果媒体数据不只是保存在本地文件里，而要跨设备发送，那系统又会进入另一层：

网络传输层。

这时你面对的问题，不再只是“文件怎么存”，而变成：

  • 怎么发
  • 怎么收
  • 怎么抗抖动
  • 怎么处理丢包
  • 怎么保证实时性

8.1 这层常见会碰到什么

比如：

  • RTP / RTCP
  • RTSP
  • RTMP
  • WebRTC

8.2 为什么网络层不是简单“把 MP4 发出去”

因为网络传输关注的是：

  • 包怎么切
  • 时序怎么对齐
  • 延迟怎么控制
  • 丢包怎么补救
  • 实时性和稳定性怎么平衡

也就是说，到这一步，媒体数据已经不只是“内容文件”，而变成了一种：

面向实时传输和接收的流式数据。

这也是为什么很多人一进入流媒体领域，会突然觉得问题复杂度激增。

因为网络一接进来，系统就从“本地处理问题”变成了“分布式时序问题”。

九、第七层：接收端为什么还要再走一遍“拆”和“还原”

当数据从文件或网络来到接收端时，系统不会直接把它送去显示。

它还要走一个“反向还原”的过程。

9.1 如果是封装数据，要先 demux

比如 MP4、TS、FLV 这些容器，接收端要先把里面的音视频流拆出来。

也就是：

  • 把视频流拿出来
  • 把音频流拿出来
  • 恢复出各自的压缩 packet

9.2 如果是网络传输数据，要先收流解析

比如 RTP / RTMP / WebRTC 收到的数据，要先经过协议层解析，才能恢复出真正的压缩媒体数据。

9.3 然后再解码

接下来才是：

  • 视频码流 -> 解码成原始视频帧
  • 音频码流 -> 解码成原始音频采样帧

也就是说，接收端其实是在做一件事：

把之前为了存储和传输做过的压缩组织，再一步步拆回来。

十、第八层：渲染与播放，为什么“解码完了”还不等于“结束了”

很多人学音视频时，容易下意识把“解码”当成终点。

但其实不是。

10.1 视频解码后为什么还不能直接显示

因为视频解码出来的 frame，通常还是工程内部更适合处理的数据，比如：

  • YUV420P
  • NV12
  • 特定分辨率
  • 特定颜色空间

而最终上屏往往还需要：

  • 缩放
  • 色彩空间转换
  • 渲染到 GPU / 显示层
  • 和显示刷新节奏同步
  • OSD / UI 叠加

10.2 音频解码后为什么也还没结束

音频解码后，同样还需要：

  • 重采样
  • 声道处理
  • 音量处理
  • 送给播放设备

也就是说，解码只是把“压缩表达”还原回“可处理内容”，而不是最终终点。

最后真正面向用户的，是：

  • 屏幕上的画面
  • 扬声器里的声音

十一、把整条系统压成一句更像工程师的话，应该怎么说

如果你真正理解了上面这些层次，那你可以把音视频系统概括成这样一句话：

音视频系统本质上是在不同层之间搬运和转换媒体数据，它围绕原始采样数据、原始帧、压缩码流、封装数据、网络流和最终渲染输出这几种核心形态，完成采集、处理、压缩、传输、还原和播放。

这句话比单纯说“音视频系统就是采集编码播放”要强很多。

因为它体现了两件事：

  1. 你知道系统里真正处理的对象是什么
  2. 你知道这些对象会在不同层之间不断变化

这就是系统认知。

十二、为什么这个视角会直接帮助你理解后面的所有主题

一旦你把数据形态这件事想清楚，后面很多看起来复杂的主题都会变顺。

12.1 为什么 FFmpeg 里有 AVPacket 和 AVFrame

因为它们本来就在对应不同的数据形态：

  • AVFrame 更偏原始内容帧
  • AVPacket 更偏压缩后的组织单位

12.2 为什么 filter 一般处理的是 frame，而不是码流

因为滤镜通常处理的是内容本身，而不是压缩表达。

12.3 为什么时间戳这么麻烦

因为数据一旦跨过：

  • 编码
  • 封装
  • 网络传输
  • 解码
  • 渲染

就会有多套时间语义同时存在。

12.4 为什么 Camera、编码、协议、显示明明看着是 4 件事，本质上却能串起来

因为它们都在处理：

同一份媒体数据在不同层的不同存在形式。

只不过关注点不同。

十三、面试里怎么讲这个问题，才不像在背定义

如果面试官问你：

你怎么理解音视频系统到底在处理什么？

不建议回答成：

  • 它处理视频和音频
  • 它会采集、编码、解码、播放

这太平了。

更好的讲法应该是：

13.1 先讲本质

音视频系统本质上是在处理媒体数据在不同层之间的形态转换，不同阶段的数据形式并不一样。

13.2 再举核心层次

比如它会涉及原始采样数据、原始 frame、压缩码流、封装后的文件数据、网络传输数据，以及最终解码后用于渲染和播放的内容帧。

13.3 最后讲意义

只有把这些数据形态区分清楚，后面理解编码、封装、传输、FFmpeg、显示和同步问题，才不会混在一起。

如果你能这么回答，面试官一般会明显感觉到：

你脑子里是有一张系统图的。

总结：音视频系统真正处理的，不是“一个视频”，而是一连串不断变形的媒体数据

回到这篇文章最开始的问题：

音视频系统到底在处理什么？

我的理解是：

它处理的不是一个静态的“视频文件”，而是一整串在不同阶段不断变化形态的媒体数据。

这串变化大致会经历：

  • 现实世界信号
  • 采样后的原始数据
  • 原始 frame
  • 压缩码流
  • 封装 / 网络流数据
  • 接收端解析后的媒体数据
  • 解码后的 frame
  • 最终渲染与播放输出

而音视频工程真正做的，就是围绕这些形态完成：

  • 采
  • 处
  • 编
  • 传
  • 解
  • 显

只要这条主线立住，后面不管是：

  • Camera Pipeline
  • H.264 / H.265
  • AAC
  • RTP / RTSP / WebRTC
  • FFmpeg
  • OpenGL
  • 弱网优化
  • 音画同步

你都会更容易找到它在整个系统里的位置。

这也是我觉得学习音视频时，最值得先建立的底层认知之一：

始终先问自己，当前面对的数据，究竟处于哪一种形态。
